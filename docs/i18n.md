# Localization (i18n)

A lightweight, dependency-free i18n layer lives at the end of
`frontend/js/app.js` (loaded on every page), plus a language switcher injected
into the nav site-wide. The chosen language persists in `localStorage` under
`4k_lang` and sets `<html lang>`.

## How translation is applied

Tag any element with one of:

| Attribute | Effect |
|---|---|
| `data-i18n="key"`      | sets `textContent` |
| `data-i18n-html="key"` | sets `innerHTML` (for strings with inline markup, e.g. the hero title) |
| `data-i18n-ph="key"`   | sets the `placeholder` |

`window.applyI18n(root)` walks those attributes and substitutes the active
language; it runs on load, after the footer/newsletter are injected, and on
every language change. For strings built in JS use `window.t('key')`.

Currently tagged: the index nav, hero, and search card, plus the site-wide
newsletter band. Extend coverage by adding `data-i18n` attributes to more
markup (and the matching keys to the dictionary) — no other wiring needed.

## Adding a language

In the `I18N` map (in the i18n IIFE in `app.js`) add a new language block with
the same keys, and add its code to `LANGS` so it appears in the switcher:

```js
const LANGS = { en: 'EN', fr: 'FR', yo: 'YO' };  // add code + short label
const I18N = {
  en: { 'nav.home': 'Home', /* … */ },
  fr: { 'nav.home': 'Accueil', /* … */ },
  yo: { 'nav.home': 'Ilé', /* … */ },             // new block, same keys
};
```

Missing keys fall back to English automatically, so a partial translation is
safe to ship. Shipped today: **English (default)** and **French**.
